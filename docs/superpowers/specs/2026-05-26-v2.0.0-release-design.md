## The 429 fix

**File:** `custom_components/bticino_intercom/coordinator.py`

**Root cause:** Netatmo's push WebSocket broadcasts the same RTC
`terminate`/`rescind` event to every device registered for push on the
account, each with a distinct `_id` push token but the same
`session_id`. The current code dedups `offer` (lines 376-380) but NOT
`terminate`/`rescind`. Each duplicate triggers `async_request_refresh()`
→ 3 HTTP calls. 6 duplicates in 2s = ~18 calls → 429.

**Fix:**

```python
def _end_call_session(self, module_id: str, reason: str) -> bool:
    """Close active call session. Returns True if a session was actually
    closed, False if it was already closed (duplicate event)."""
    session = self._active_calls.pop(module_id, None)
    if session is None:
        return False
    watchdog = session.get("watchdog")
    if watchdog is not None and not watchdog.done():
        watchdog.cancel()
    _LOGGER.debug("Call session for module %s closed (reason=%s)", module_id, reason)
    return True

# in _process_rtc_event, terminate branch:
if rtc_action == "terminate":
    dedup_id = calling_module_id or display_id
    closed = self._end_call_session(dedup_id, reason="terminate")
    if not closed:
        _LOGGER.debug(
            "Ignoring retransmitted terminate for module %s (session already closed)",
            dedup_id,
        )
        return False  # ← no refresh, no logbook spam

    # ...existing logic (signal dispatch, fire event, logbook, history)
    return True

# same shape for rescind branch
```

**Test (TDD, written first):**

```python
async def test_terminate_duplicates_trigger_single_refresh(hass, coordinator):
    """6 identical terminate events for the same session_id must trigger
    exactly 1 async_request_refresh call (not 6)."""
    # given: an active call session
    coordinator._active_calls["module-x"] = {"session_id": "abc", ...}

    # when: 6 duplicate terminate messages arrive
    msg = build_terminate_message(session_id="abc", module_id="module-x")
    with patch.object(coordinator, "async_request_refresh") as refresh:
        for _ in range(6):
            await coordinator._handle_websocket_message(msg)

    # then: refresh called once (first terminate), not 6 times
    assert refresh.call_count == 1
```

**Bonus effect:** the duplicate `bticino_intercom_terminated` logbook
events (10 per call in our DB) also collapse to one.

## Release notes draft

See [release-notes-v2.0.0.md](#release-notes-draft) inline below.

The notes use the rc12 structure (proven to read well) and:
- emoji only in section headings, never inline
- "Stability" section absorbs the 429 fix — it was a bug fix, not a feature

### Release notes draft

```markdown
# v2.0.0 — WebRTC live video & two-way audio

After 12 release candidates and ~6 weeks of beta, v2.0 is stable.

## What's new since v1.9.x

### 🎥 WebRTC live video (multi-camera)
One camera entity per external unit (BNEU). Live video on demand from the
HA dashboard, with two-way audio when paired with the custom Lovelace card
from [bticino_ha_extras]. Chrome/Chromium only — see [Firefox investigation].

### 📞 Answer mode
Pick up an incoming call straight from HA: the device's SDP offer (sent
via push WebSocket) is answered with an SDP from the browser.

### 🗂️ Call history with persistent snapshots
Snapshots and vignettes are downloaded to disk the moment the call arrives,
so they remain viewable after the Azure SAS URLs expire. Browse via HA
Media Source or a dedicated authenticated HTTP view. Configurable retention
(default 30 days / 500 events).

### 🔔 Doorbell event entity (HA standard)
Adds `event.<intercom>_doorbell` using `DoorbellEventType.RING` — works
with HA's generic `doorbell.rang` trigger. The existing `binary_sensor`
stays (dual-entity pattern, same approach UniFi uses).

### 🏠 Call Home camera
Voice-only intercoms (no camera) get a "Call Home" camera entity with
a SEELE-style poster so the answer-mode UI still works.

### 📦 Blueprints
- `actionable_doorbell_notification` — Answer / Open / Reject buttons
- `incoming_call_notification` — with ringtone

### 🛡️ Stability
- 429 rate-limit prevention: terminate/rescind RTC retransmits are now
  deduplicated by session_id (fix for #56)
- Proactive WebSocket re-subscribe + resilient coordinator
- Smart retry backoff at boot

## Upgrade requirements
- Home Assistant ≥ **2026.5.0**
- pybticino `>=1.8.2`
- For two-way audio: [bticino_ha_extras] custom card (Chrome only)

For HA < 2026.5: stay on v1.9.x.

## Breaking changes
None for users on v1.9.x with default config. The event entity is new
(additive). DoorbellEventType is wire-compatible (still emits "ring").
```

## Out of scope

## Risks

- **rc13 soak with no real call:** if no doorbell rings during the soak
  window, the 429 fix only has unit-test coverage. Acceptable risk — the
  fix is small, localized, and the test exercises the exact pathway.
- **Merge conflicts on README:** `main` is at v1.9.7, `dev/webrtc` may
  have older README copy. Resolve in favor of `dev/webrtc` then patch
  the "stable" hint as Step 6.
- **CI on the merge commit:** if HACS validation flags anything new
  (manifest version, hacs.json HA minimum), we catch it in the PR
  before tagging stable.
